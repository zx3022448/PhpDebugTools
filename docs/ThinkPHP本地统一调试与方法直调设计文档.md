# ThinkPHP 本地统一调试与方法直调设计文档

## 1. 背景

目标是开发一个面向 PhpStorm 的 PHP 调试插件，重点解决 ThinkPHP 项目在本机环境下的调试体验问题，尤其包括以下几类场景：

1. 本地 Web 请求可以稳定进入断点。
2. 本地 CLI 请求可以稳定进入断点。
3. 可以在 IDE 中直接选择控制器方法或服务类方法并传入参数执行。
4. 支持静态方法与普通实例方法的直调。
5. 提供常见断点失效、Xdebug 配置错误、运行环境不一致等问题的诊断能力。

## 2. 当前已确认前提

根据本轮需求沟通，当前设计基于以下前提：

1. 主要目标环境为 `纯本机 Windows`。
2. 第一目标框架为 `ThinkPHP`。
3. 第一版优先支持 `控制器方法` 与 `服务类方法`。
4. 插件允许在目标项目根目录下自动创建 `.php-debug-tools/` 辅助目录。

这些前提决定了第一版设计不以 Docker、WSL、远程服务器为主要目标，也不以多框架统一抽象为第一优先级。

## 3. 设计目标

第一阶段希望达到的效果如下：

1. 用户在 PhpStorm 中打开 ThinkPHP 项目后，插件能够识别项目并给出调试能力入口。
2. 用户可以通过插件发起可控的本地 Web 调试或 CLI 调试，并尽量稳定地命中断点。
3. 用户可以选择一个控制器方法或服务类方法，输入参数后直接执行。
4. 调试失败时，插件能够给出明确的诊断提示，而不是只让用户自己排查 Xdebug。
5. 后续可在不推翻架构的前提下扩展到 ThinkPHP 5/6 细分适配、多框架支持和更强的自动调试接管能力。

## 4. 非目标

第一版明确不做以下事情：

1. 不承诺接管“所有任意来源的本地请求”并无条件自动进断点。
2. 不优先支持 Docker、WSL、远程主机或容器内部 PHP 运行环境。
3. 不在第一版就覆盖 Laravel、Symfony、WordPress 等其他框架。
4. 不在第一版构建复杂的本地守护进程或系统级代理程序。

## 5. 方案对比

### 方案 A：纯 IDE 无侵入增强

由插件提供配置检查、Xdebug 诊断、CLI/Web 启动辅助和断点失败排查，但不在项目内写入辅助运行时代码。

优点：

1. 实现快。
2. 对项目侵入低。
3. 兼容性风险较小。

缺点：

1. 很难真正统一 Web 与 CLI 调试链路。
2. 方法直调能力受限。
3. 很难把参数注入、请求模拟、框架引导做得稳定。

### 方案 B：IDE 插件 + 项目内辅助运行时

插件层负责界面、项目识别、调试编排和诊断；项目根目录下的 `.php-debug-tools/` 负责真实执行 ThinkPHP 上下文内的请求模拟与方法调用。

优点：

1. 能把 Web 调试、CLI 调试、方法直调统一到一个受控执行模型中。
2. 更适合 ThinkPHP 这类依赖框架引导和容器上下文的项目。
3. 便于后续支持 ThinkPHP 5/6 差异适配。
4. 更接近用户提出的真实目标。

缺点：

1. 需要维护项目内辅助脚本。
2. 需要处理插件版本与运行时版本同步问题。
3. 需要设计好执行边界与安全提示。

### 方案 C：本地调试代理/拦截器

在插件之外再增加一个本地守护进程或代理层，尝试统一控制 PHP 请求和调试会话。

优点：

1. 理论上能力最强。

缺点：

1. 工程复杂度明显过高。
2. Windows 本机环境下稳定性与维护成本风险很高。
3. 第一版极易陷入基础设施建设而迟迟无法形成可用产品。

## 6. 推荐方案

推荐采用 `方案 B：IDE 插件 + 项目内辅助运行时`。

推荐原因：

1. 它最贴近“统一调试入口 + 方法直调”的核心诉求。
2. 它能在不引入额外守护进程的前提下，把 PHP 真正放回目标项目上下文执行。
3. 它为后续扩展 ThinkPHP 版本兼容、调试诊断增强和更多执行模式预留了自然空间。

## 7. 整体架构

整体采用双层结构：

1. `PhpStorm 插件层`
2. `项目内调试运行时层`

### 7.1 PhpStorm 插件层职责

插件层只负责 IDE 侧能力，不直接承担 PHP 业务执行逻辑。

核心职责包括：

1. 识别当前项目是否为 ThinkPHP 项目。
2. 检测本机 PHP、Xdebug、php.ini、端口、IDE Key 等调试环境。
3. 创建与同步 `.php-debug-tools/` 运行时目录。
4. 提供统一的调试入口，包括 CLI 调试、Web 调试、方法直调。
5. 提供方法选择、参数编辑和执行控制界面。
6. 展示本次执行状态、调用信息与错误诊断结果。

### 7.2 项目内调试运行时层职责

运行时层位于项目根目录 `.php-debug-tools/`，负责在真实 PHP 项目上下文内执行请求与方法调用。

核心职责包括：

1. 加载 Composer 自动加载。
2. 初始化 ThinkPHP 应用。
3. 构造请求上下文或方法调用上下文。
4. 调用控制器方法、服务类方法或静态方法。
5. 输出执行结果、异常、诊断信息和结构化状态。

## 8. 插件层核心模块

### 8.1 环境诊断模块

用于检查以下内容：

1. PHP CLI 是否可用。
2. Xdebug 是否已加载。
3. Xdebug 关键配置是否合理。
4. Web 与 CLI 使用的 PHP 配置是否存在明显不一致。
5. 常见端口冲突或 IDE 监听问题。
6. 常见断点失效原因，例如路径映射错误、未触发调试会话、脚本未走目标入口等。

### 8.2 项目识别模块

负责识别：

1. 当前项目是否为 ThinkPHP 项目。
2. 更接近 ThinkPHP 5 还是 ThinkPHP 6。
3. 入口文件、应用目录、Composer 信息和关键配置特征。

### 8.3 运行时同步模块

负责将插件内置的运行时模板同步到项目中的 `.php-debug-tools/` 目录，并维护：

1. 运行时版本号。
2. 生成文件清单。
3. 配置文件。
4. 升级策略。

### 8.4 调试编排模块

统一发起三类动作：

1. `CLI 调试`
2. `Web 调试`
3. `方法直调`

该模块负责拼装参数、选择入口脚本、附加调试环境变量并触发执行。

### 8.5 方法选择与参数编辑模块

负责：

1. 在 IDE 中选择控制器方法或服务类方法。
2. 解析方法签名。
3. 生成参数输入表单。
4. 支持基础类型、数组、JSON 文本等参数输入方式。
5. 支持静态方法和实例方法。

### 8.6 结果与状态面板

用于展示：

1. 本次调用的目标类、方法、参数和执行入口。
2. 调试会话是否已触发。
3. 运行时返回的标准输出、异常信息和结构化错误。
4. 失败时的修复建议。

## 9. `.php-debug-tools/` 运行时目录建议结构

建议初版包含以下文件：

1. `bootstrap.php`
2. `invoke-controller.php`
3. `invoke-service.php`
4. `debug-web-entry.php`
5. `runtime-config.json`

### 9.1 `bootstrap.php`

负责统一完成：

1. 加载 Composer 自动加载。
2. 探测 ThinkPHP 版本或入口路径。
3. 初始化应用上下文。
4. 提供通用工具函数与错误捕获。

### 9.2 `invoke-controller.php`

负责直接调用控制器方法，支持模拟：

1. 路由参数
2. Query 参数
3. POST 参数
4. Header
5. Cookie

### 9.3 `invoke-service.php`

负责调用服务类方法，支持：

1. 实例方法调用
2. 静态方法调用
3. 参数反序列化
4. 必要时通过容器解析实例

### 9.4 `debug-web-entry.php`

作为专用本地调试入口，用于将 Web 请求导入受控调试链路，以提高断点命中稳定性。

### 9.5 `runtime-config.json`

用于记录：

1. 项目识别结果
2. 入口文件信息
3. 默认执行模式
4. 参数序列化策略
5. 运行时版本信息

## 10. 第一版重点能力边界

第一版建议只保证以下能力：

1. 支持 ThinkPHP 项目识别。
2. 支持本机 PHP CLI 环境诊断。
3. 支持受控发起的本地 CLI 调试。
4. 支持受控发起的本地 Web 调试。
5. 支持控制器方法直调。
6. 支持服务类方法直调。
7. 支持静态方法与普通实例方法。

其中最重要的产品原则是：

**先保证“从插件发起的请求能稳定进入断点”，再逐步增强到更广义的“本地请求自动进入断点”。**

## 11. 风险与注意事项

### 11.1 ThinkPHP 版本差异

ThinkPHP 5 与 6 在应用引导、容器、请求处理和目录结构上存在差异。第一版应先建立兼容层，不应过早抽象成统一无差别模型。

### 11.2 直调方法的副作用

某些服务方法可能依赖完整请求上下文、数据库事务、登录态或全局单例状态。插件需要对“非纯函数方法直调”给出提示，避免误导用户。

### 11.3 调试触发条件并非完全由插件单方面决定

即使插件完成执行编排，断点是否命中仍与 Xdebug 配置、IDE 监听状态、PHP 实际加载的配置文件和入口路径有关，因此诊断模块是核心能力，不是附属能力。

### 11.4 项目内辅助目录需要可管理

`.php-debug-tools/` 必须具备明确的版本标记、可重建能力和忽略提交建议，避免污染业务代码仓库。

## 12. 下一步设计建议

在本设计基础上，后续应继续细化以下内容：

1. 本地 Web 请求与 CLI 请求如何稳定触发调试会话。
2. 控制器方法直调链路如何构造 ThinkPHP 请求上下文。
3. 服务类方法直调链路如何做容器解析与参数绑定。
4. ThinkPHP 5 与 ThinkPHP 6 的识别与适配策略。
5. PhpStorm 插件侧的 UI 入口与交互流程。

## 13. 当前结论

本需求不适合做成单一面板或简单 Xdebug 配置助手，而更适合做成一套面向 ThinkPHP 的“本地统一调试工作台”。

第一版最合理的落点是：

1. 以 ThinkPHP 为第一目标框架。
2. 以本机 Windows 环境为第一目标运行场景。
3. 采用 `PhpStorm 插件 + 项目内运行时` 的双层结构。
4. 先打通 `受控请求调试 + 方法直调` 两条核心链路。

## 14. 核心链路设计

第一版应围绕两条核心链路构建：

1. `受控请求调试链路`
2. `方法直调链路`

两条链路在入口不同，但底层协议应统一为：

`插件收集调用信息 -> 写入运行时配置 -> 选择入口脚本 -> 附加调试触发条件 -> 执行 -> 回传结果`

### 14.1 受控请求调试链路

第一版不承诺接管任意来源的全部本地请求，而是先提供两个稳定、可控的标准调试入口。

#### CLI 调试入口

由插件调用本机 PHP，并显式附加调试所需环境变量或运行参数，然后进入 `.php-debug-tools/` 的 CLI 入口脚本。

该链路的特点：

1. 稳定性最高。
2. 最容易验证。
3. 最容易做故障定位。

#### Web 调试入口

由插件生成专用本地调试 URL，请求先进入 `.php-debug-tools/debug-web-entry.php`，再导入 ThinkPHP 正式应用入口。

该链路的作用：

1. 统一调试触发条件。
2. 统一请求参数、Header、Cookie 和路由信息注入方式。
3. 提高本地页面调试的断点命中可控性。

### 14.2 方法直调链路

第一版只聚焦两类调用目标：

1. `控制器方法`
2. `服务类方法`

#### 控制器方法直调

控制器方法直调更接近一次受控的应用请求。插件侧采集：

1. 路由参数
2. Query 参数
3. POST 参数
4. Header
5. Cookie

运行时侧负责将这些数据组装成 ThinkPHP 可识别的请求上下文，再调用目标 action。

#### 服务类方法直调

服务类方法直调更适合业务逻辑调试。插件侧解析方法签名并生成参数输入区，运行时侧区分：

1. 实例方法
2. 静态方法

对于实例方法，优先通过 ThinkPHP 容器解析对象；如无法解析，再给出明确错误。

### 14.3 参数传递策略

第一版建议统一使用 `结构化 JSON + 简单类型推断` 的方式。

规则建议如下：

1. 标量参数按方法签名转换为 `int`、`bool`、`float`、`string`。
2. 数组和复杂结构通过 JSON 文本输入。
3. 控制器请求上下文单独放在 `request` 结构中。
4. 服务方法参数放在 `args` 数组中并保持顺序。

第一版暂不支持复杂对象参数自动构造。

## 15. ThinkPHP 5/6 识别与兼容策略

### 15.1 识别策略

不建议仅依赖单一特征判断版本，而应采用多信号综合识别。

建议按以下顺序检查：

1. `composer.json` 中 `topthink/framework` 的版本约束。
2. `vendor` 中已安装包的实际版本信息。
3. 目录结构特征。
4. `public/index.php` 等入口文件的引导写法。

最终识别结果建议分为：

1. `明确为 ThinkPHP 5`
2. `明确为 ThinkPHP 6`
3. `疑似 ThinkPHP，但版本不确定`

如果版本不确定，不应默认开放全部能力，而应提示用户确认项目入口与版本信息。

### 15.2 兼容实现策略

不建议在一个超大 `bootstrap.php` 中处理全部版本差异，而建议采用：

1. 公共入口层
2. 版本适配层

建议结构如下：

1. `bootstrap.php`
2. `adapters/thinkphp5.php`
3. `adapters/thinkphp6.php`

其中：

1. `bootstrap.php` 负责读取配置、统一异常处理、选择适配器。
2. `thinkphp5.php` 负责 ThinkPHP 5 的应用引导与调用。
3. `thinkphp6.php` 负责 ThinkPHP 6 的应用引导与调用。

### 15.3 控制器直调兼容原则

控制器直调应尽量通过框架自身的应用、容器和请求链路完成，而不是简单反射调用。

第一版至少应保证：

1. 应用正确引导。
2. 请求对象可用。
3. 控制器实例来自框架容器或等价构造逻辑。

### 15.4 服务类直调兼容原则

服务类直调建议按以下优先级执行：

1. 实例方法优先通过框架容器解析。
2. 如容器无法解析，再尝试无参构造。
3. 如仍失败，返回明确错误与修复建议。

静态方法则在完成类与方法存在性校验后直接调用。

## 16. 插件 UI 与用户操作流设计

### 16.1 入口设计

第一版建议提供 `3 个入口 + 1 个主工作台`：

1. `工具窗口`
2. `编辑器右键 / 类方法右键`
3. `Project 视图右键`
4. `主工作台`

说明如下：

1. 工具窗口用于承载环境状态、调试入口、结果面板和诊断能力。
2. 方法右键用于快速发起“调试此方法”。
3. 项目右键用于初始化或更新 `.php-debug-tools/` 及执行环境诊断。

第一版不强依赖 Gutter 图标，可后续迭代补充。

### 16.2 工具窗口结构

建议按 Tab 划分，而不是单页堆叠：

1. `概览`
2. `请求调试`
3. `方法直调`
4. `诊断`

其中：

1. `概览` 展示项目识别结果、ThinkPHP 版本推断、PHP/Xdebug 状态、运行时同步状态。
2. `请求调试` 承载 CLI 和 Web 两类受控调试入口。
3. `方法直调` 承载类方法选择、参数编辑和执行动作。
4. `诊断` 展示失败原因分层、日志和修复建议。

### 16.3 请求调试操作流

建议流程如下：

1. 用户进入 `请求调试`。
2. 选择 `CLI` 或 `Web`。
3. 填写必要参数。
4. 点击 `开始调试`。
5. 插件自动检查监听状态、运行时状态和参数完整性。
6. 发起请求并展示结果。

默认应尽可能复用最近一次成功配置，以降低重复输入成本。

### 16.4 方法直调操作流

建议流程如下：

1. 用户在控制器方法或服务类方法上右键。
2. 选择 `调试此方法`。
3. 弹出参数面板。
4. 用户填写参数。
5. 点击 `执行并调试`。
6. 插件将调用信息交给 `.php-debug-tools/` 执行。
7. 命中断点后回到标准 PhpStorm 调试体验。

参数面板建议拆为两部分：

1. `基础参数区`
2. `请求上下文区`

其中 `请求上下文区` 仅在控制器方法直调时显示。

### 16.5 最近记录与复用

建议第一版就支持以下最近记录能力：

1. 最近调试过的 URL
2. 最近调试过的方法
3. 最近一次输入的参数
4. 最近一次成功配置

## 17. 错误处理、诊断分层与测试策略

### 17.1 诊断分层

所有失败应尽量归类到以下四层之一：

1. `IDE 层`
2. `PHP / Xdebug 层`
3. `框架引导层`
4. `目标调用层`

分层说明：

1. `IDE 层` 包括监听状态、插件配置、运行时同步状态等。
2. `PHP / Xdebug 层` 包括 CLI 路径、扩展加载、端口、触发条件等。
3. `框架引导层` 包括 Composer、入口文件、应用初始化和版本适配失败等。
4. `目标调用层` 包括类不存在、方法不存在、参数绑定失败、容器解析失败和业务异常。

### 17.2 统一返回格式

建议 `.php-debug-tools/` 的所有入口脚本都返回统一结构化结果，而不是零散文本输出。

统一字段建议至少包括：

1. `status`
2. `stage`
3. `message`
4. `details`
5. `hints`

这样插件端可以稳定展示、记录和映射错误类型。

### 17.3 静态诊断与动态诊断

建议同时具备两类诊断：

1. `静态诊断`
2. `动态诊断`

其中：

1. 静态诊断用于检查配置、版本、路径、入口和环境状态。
2. 动态诊断用于执行一次最小调用，验证调试链路能否真实拉起。

### 17.4 方法直调失败提示原则

对方法直调失败，应尽量区分：

1. 容器服务无法解析。
2. 目标方法依赖请求上下文。
3. 参数数量或类型不匹配。
4. 业务代码自身抛出异常。

不应把所有问题都笼统包装为“插件执行失败”。

### 17.5 测试策略

建议分三层测试：

1. `插件层单元测试`
2. `运行时层集成测试`
3. `端到端验证`

测试重点包括：

1. ThinkPHP 5/6 识别准确性。
2. 运行时同步与升级是否稳定。
3. 控制器方法与服务方法参数绑定是否正确。
4. Xdebug 未加载时是否能准确诊断。
5. 容器解析失败时是否能清晰归因。

## 18. 第一版 MVP 范围与开发优先级

### 18.1 MVP 核心价值

第一版只聚焦三类核心价值：

1. 解释“为什么进不了断点”。
2. 提供稳定的受控请求调试入口。
3. 提供可传参的方法直调能力。

### 18.2 MVP 必做范围

建议第一版必须包含：

1. ThinkPHP 项目识别。
2. 环境诊断。
3. 运行时同步。
4. CLI 调试入口。
5. Web 调试入口。
6. 控制器方法直调。
7. 服务类方法直调。
8. 统一结果面板。

### 18.3 MVP 暂缓范围

建议第一版暂不包含：

1. 自动接管所有浏览器请求。
2. 复杂对象参数自动构造。
3. 多框架支持。
4. Gutter 图标与大量快捷入口。
5. 请求录制与回放。
6. Docker、WSL、远程环境支持。

### 18.4 推荐开发顺序

建议按如下顺序实施：

1. `环境诊断 + ThinkPHP 识别`
2. `运行时目录同步`
3. `CLI 调试链路`
4. `服务类方法直调`
5. `控制器方法直调`
6. `Web 调试链路`

这个顺序的目的是先从最稳定、最可验证的能力开始，逐步搭建更复杂的链路。

### 18.5 MVP 验收标准

建议以以下结果判断第一版是否达标：

1. 插件能识别 ThinkPHP 项目并展示环境状态。
2. 用户能一键执行本地 CLI 调试并命中断点。
3. 用户能选择服务类方法并带简单参数执行调试。
4. 用户能选择控制器方法并带请求参数执行调试。
5. 失败时插件能明确说明问题位于哪一层，并给出修复建议。
