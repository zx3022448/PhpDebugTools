# 工具窗口 UI 参考页对齐需求文档

## 1. 背景

当前 `PHP Debug Tools` 插件的工具窗口已经具备方法搜索、PHP 运行时选择、控制器请求上下文编辑、服务方法参数编辑和执行结果展示等核心能力，但整体 UI 仍以传统 Swing 表单式布局为主。

用户提供了 `C:\Users\PC\Desktop\untitled-2-html` 作为新的目标界面，希望插件工具窗口在视觉风格、信息架构和分区节奏上尽量与该参考页面保持一致，而不是仅做局部换色或轻量卡片化处理。

本次改版的核心目标不是新增功能，而是将现有能力重组为更接近参考页的 Apple 风格调试工作台。

## 2. 目标

本次改版需要同时满足以下目标：

1. 工具窗口整体结构尽量贴近参考 HTML 的布局和视觉层级。
2. 在不改变现有方法直调逻辑的前提下，完成“壳层复刻 + 业务组件嵌入”。
3. 将方法选择、请求编辑、状态查看和结果查看组织成更连续的一屏工作流。
4. 让浅色主题下的视觉气质接近参考页面的玻璃拟态工作台。
5. 在 Swing 可维护性范围内尽量还原参考页的标题栏、品牌区、最近执行区、请求工作台和结果侧栏。

## 3. 非目标

本次改版明确不包含以下内容：

1. 不新增新的业务功能或新的调试协议。
2. 不修改方法执行器、运行时脚本、请求组装逻辑和缓存逻辑。
3. 不将 Swing UI 替换为内嵌浏览器页面。
4. 不追求网页级像素级复刻，包括真实毛玻璃模糊和 CSS 动画细节。
5. 不一次性重绘插件所有页面，本轮重点仍是工具窗口与方法直调页。

## 4. 参考页特征总结

`C:\Users\PC\Desktop\untitled-2-html` 中的目标页面具有以下稳定特征：

1. 最外层是带圆角的拟态窗口壳，顶部有 macOS 风格标题栏。
2. 标题栏下方存在品牌区、导航 chips 和最近执行横向卡片区。
3. 主内容区分为左侧大工作台和右侧结果侧栏两列结构。
4. 左侧大工作台内部先是标题和环境标签，再是方法选择与请求配置，再进入分段式内容区。
5. 右侧侧栏由状态卡、指标卡和深色输出面板组成。
6. 整体视觉语言以浅色玻璃拟态卡片、细边框、胶囊按钮和深色代码面板为主。

这些特征将作为本轮插件 UI 重组的主要参照物。

## 5. 设计范围

### 5.1 本次重点

本次重点改造以下区域：

1. `PhpDebugToolsToolWindowPanel`
2. `MethodInvokeToolWindowPanel`
3. `ControllerRequestEditorPanel`
4. `RequestParameterTablePanel`
5. `ToolWindowUiStyles`

### 5.2 低优先级范围

`概览` 页仅做最小程度的视觉统一，不做和参考页同级别的完整结构复刻。

## 6. 总体实现策略

本次采用已经确认的 `C` 方案：

> 近似 1:1 复刻参考页面结构，再把现有插件业务组件嵌进去；只在 Swing 无法完美表达的细节上做视觉近似。

实现原则如下：

1. 外层结构优先对齐参考页的信息架构。
2. 内层业务组件优先复用现有逻辑和 Swing 组件。
3. 视觉系统通过统一样式支撑层集中管理，不在各个面板里分散硬编码。
4. 为降低回归风险，尽量重组组件位置，而不是重写组件行为。

## 7. 信息架构设计

### 7.1 最外层窗口壳

工具窗口整体重组为拟态工作台外壳，包含以下区域：

1. 假标题栏
2. 品牌与导航区
3. 最近执行横向卡片区
4. 方法直调主工作区

其中标题栏和品牌区承担“参考页面识别感”的职责，最近执行区承担快速回顾最近调试上下文的职责。

### 7.2 标题栏

标题栏需要模拟参考页中的桌面窗口感觉，包含以下元素：

1. 左侧窗口控制点占位
2. 中部工具窗口标题与模式标签
3. 右侧轻量操作按钮

说明：

1. 这些元素主要承担视觉对齐作用，不要求模拟真实系统窗口行为。
2. 标题栏不承载复杂业务交互。

### 7.3 品牌与导航区

标题栏下方新增品牌与导航区，包含：

1. 左侧插件品牌卡
2. 右侧导航 chips
3. 下方最近执行横向卡片

导航 chips 的目标不是新增复杂导航逻辑，而是将现有 `概览 / 方法直调` 两个入口表达得更接近参考页；如果结构允许，可预留 `运行时` 视觉入口但不新增业务 tab。

### 7.4 最近执行区

最近执行区使用横向小卡片展示最近调试记录，优先使用现有 `RecentDebugStore` 中已可取得的数据。

建议展示：

1. 方法签名
2. 最近状态语义
3. 简短时间信息

如果当前状态数据不完整，允许使用保守占位表达，但不能伪造不存在的真实执行结果。

### 7.5 主工作区

主工作区采用两列布局：

1. 左侧请求工作台
2. 右侧状态与结果侧栏

这是本轮最重要的结构调整。

## 8. 左侧请求工作台设计

### 8.1 工作区头部

左侧工作台顶部显示：

1. 小号分类标签
2. 大号标题，如“方法直调工作区”
3. 环境胶囊标签，如 PHP 版本或 ThinkPHP 识别结果

这部分对应参考页中 `Method Workbench` 与环境 pill 的结构。

### 8.2 请求工作台主卡

左侧主体使用一张大卡片承载方法选择和请求配置，内部自上而下包含：

1. 分区标题与简要说明
2. 顶部操作行
3. section tabs
4. 对应 section 的内容卡

### 8.3 顶部操作行

顶部操作行尽量贴近参考页的四段式结构：

1. 搜索框
2. 方法候选选择
3. PHP 运行时选择
4. 执行按钮

说明：

1. `刷新方法` 和 `刷新 PHP` 作为次级动作，可放入主卡右上角或紧邻相关输入控件附近。
2. 为避免过度拥挤，可保留部分次级按钮为圆形 icon button。

### 8.4 section tabs

请求工作台的主要 section tabs 重组为更接近参考页的胶囊式导航。

优先保留并重新组织以下区块：

1. `概览`
2. `Query`
3. `Body`
4. `Headers`
5. `Args`
6. `Script`

这些区块仍由现有 `CardLayout` 和已有数据面板驱动，不改变其业务行为。

### 8.5 内容卡映射

各 section 映射如下：

1. `概览`：方法签名、方法类型、参数概览、说明信息
2. `Query`：查询参数表格
3. `Body`：Body 模式切换与对应输入区
4. `Headers`：请求头参数表格
5. `Args`：服务参数 JSON 或服务方法参数编辑器
6. `Script`：占位或现有脚本说明区域

控制器方法的请求上下文仍由 `ControllerRequestEditorPanel` 提供，服务方法参数输入仍由现有参数编辑逻辑提供。

## 9. 右侧状态与结果侧栏设计

右侧区域重组为三段：

1. 顶部状态卡
2. 中部指标卡组
3. 底部深色输出面板

### 9.1 状态卡

状态卡用于明确表达当前执行状态，至少覆盖：

1. 待执行
2. 执行中
3. 执行成功
4. 执行异常

状态表达应包括：

1. 小型状态指示点
2. 主状态标题
3. 可选的次级说明

### 9.2 指标卡组

指标卡组用于展示当前上下文的快速摘要，优先包含：

1. 项目识别结果
2. PHP 运行时信息
3. 当前方法参数数量或方法类型

这些卡片的价值在于减少用户在左侧主卡和结果区之间来回寻找上下文。

### 9.3 深色输出面板

输出面板保留深色代码区风格，承载以下内容：

1. 执行结果
2. 异常文本
3. 原始输出
4. 无返回值时的状态说明

深色输出面板是本轮视觉对齐中的关键元素，需要明显区别于左侧浅色工作台。

## 10. 样式策略

### 10.1 样式中心化

所有主要视觉规则集中到 `ToolWindowUiStyles.kt` 管理，包括：

1. 页面背景色
2. 窗口壳颜色
3. 卡片颜色
4. 细边框颜色
5. 主要强调色
6. 状态颜色
7. 胶囊按钮样式
8. 圆形图标按钮样式
9. 深色输出面板样式

### 10.2 浅色主导

本次以浅色参考页为主要目标视觉：

1. 页面底色使用轻渐变或轻层次背景
2. 大卡片使用半透明或近白色表面
3. 边框使用轻 hairline 感
4. 主强调色使用 Apple 风格蓝色

### 10.3 深色输出区保留对比

结果区保持深色背景，形成如下对比：

1. 左侧为浅色玻璃拟态工作台
2. 右下为深色代码与日志区

这种对比是参考页非常重要的视觉记忆点。

### 10.4 控件形态

控件形态按以下规则统一：

1. section tabs 使用胶囊式切换
2. icon button 使用圆形或圆角圆形
3. 主执行按钮为高强调色实心按钮
4. 次级动作使用浅底细边框按钮
5. 输入框使用浅底、细边框、内边距统一的卡片式输入风格

## 11. 现有组件映射与复用策略

为降低回归风险，本次尽量复用已有业务组件：

1. `searchField` 继续负责方法关键词输入
2. `methodComboBox` 继续负责候选方法选择
3. `phpExecutableComboBox` 继续负责 PHP 运行时选择
4. `ControllerRequestEditorPanel` 继续负责请求上下文编辑
5. `RequestParameterTablePanel` 继续负责 Query/Header/Body 表格输入
6. `ServiceMethodInputPanel` 继续负责服务参数输入
7. `resultArea` 继续作为结果文本输出核心组件
8. `RecentDebugStore` 继续提供最近方法与运行时缓存信息

重点改变的是这些组件的容器位置、层级、标题和视觉外壳，而不是其内部逻辑。

## 12. Swing 适配边界

以下细节允许做视觉近似，而不强求 1:1：

1. 真实毛玻璃 `backdrop-filter`
2. Web 级渐变与透明叠加精度
3. CSS hover 动效和缩放动画
4. 浏览器式响应式流式布局
5. 像素级圆角阴影叠层

适配原则如下：

1. 优先保证结构和层次接近参考页
2. 其次保证插件交互自然、性能稳定、代码可维护
3. 不为了外观相似而引入大面积自定义绘制或脆弱的组件 hack

## 13. 文案与资源调整

本轮允许补充或收敛少量文案资源，用于：

1. 标题栏模式标签
2. 分区标题
3. 状态标签
4. 指标卡标题
5. 空状态提示

文案必须继续写入 `MyMessageBundle.properties`，通过 `PhpDebugToolsBundle.message(...)` 访问。

## 14. 验证方式

### 14.1 自动验证

建议至少执行：

```powershell
.\gradlew.bat test
```

如需编译验证，可追加：

```powershell
.\gradlew.bat compileKotlin
```

### 14.2 运行验证

建议通过以下方式做人工验收：

```powershell
.\gradlew.bat runIde
```

重点验收项如下：

1. 标题栏、品牌区、最近执行区是否形成参考页式的外层工作台壳。
2. 左侧请求工作台是否已经从传统表单改为更接近参考页的分区结构。
3. 右侧状态卡、指标卡和深色结果区是否层次清晰。
4. 服务方法与控制器方法两种场景下，原有交互是否保持可用。
5. 请求表格、Body 模式切换和参数编辑态切换是否仍然正常。
6. 执行时状态展示是否能正确反映待执行、执行中、成功和异常。

## 15. 风险与控制

本次改版的主要风险包括：

1. `MethodInvokeToolWindowPanel` 布局重组过多，导致测试断言失效
2. 视觉容器增多后，滚动与尺寸计算变复杂
3. 右侧结果栏与底部参数编辑态的双态切换容易出现结构冲突
4. 最近执行区所需状态数据可能不足

对应控制策略：

1. 复用已有逻辑，优先改容器而不是改行为
2. 保留现有 `CardLayout` 和底部双态机制，只调整承载位置
3. 为关键结构保留轻量测试探针
4. 最近执行区允许优雅降级，不强行伪造不存在的数据

## 16. 预期结果

本次改版完成后，预期达到以下结果：

1. 工具窗口整体观感明显贴近用户提供的参考 HTML。
2. 方法直调主链路从“表单页”升级为“调试工作台”。
3. 左侧请求配置与右侧执行反馈形成更清晰的协作关系。
4. 现有功能逻辑保持不变，但视觉和结构体验显著提升。
5. 后续若继续统一概览页或其他工具窗口页面，可以复用本轮建立的样式与容器语言。
